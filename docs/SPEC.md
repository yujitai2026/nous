> 版本: v0.1.0 | 最后更新: 2026-05-12

---

## 1. 产品定位

**神智（Nous）** 是一个 to C 的多人格 AI 智能体平台。每个用户拥有五个不同性格的 AI 角色，通过持续对话，AI 会记住用户的个人信息和偏好，越用越懂你。

### 核心价值主张

- **五个懂你的 AI 伙伴** — 不是冷冰冰的工具，而是有性格、有记忆的伙伴
- **越用越智能** — AI 自动从对话中学习，积累对你的了解
- **隐私透明** — 用户可以看到 AI 记住了什么，随时可编辑或删除

### 目标用户

- 普通 C 端用户，非技术背景
- 有陪伴、咨询、闲聊需求
- 希望 AI 能记住自己、提供个性化服务

## 2. 人设定义

五个固定人设，第一版不支持用户自定义：

| ID | 名称 | 定位 | 风格 |
|---|---|---|---|
| `tech_buddy` | Code（码上聊） | 技术极客 | 专业、简洁、偶尔幽默 |
| `fitness_coach` | 铁哥 | 健身教练 | 热血、鼓励、直接 |
| `pet_expert` | 毛毛 | 宠物专家 | 温柔、耐心、专业 |
| `parenting` | 小暖 | 育儿顾问 | 温暖、共情、不说教 |
| `traveler` | 漫游 | 旅行达人 | 随性、见多识广、爱分享 |

每个人设包含：
- `id` — 唯一标识
- `name` — 显示名称
- `avatar` — 头像（emoji 或图片）
- `system_prompt` — 完整的性格设定和行为指引
- `greeting` — 首次对话的开场白

## 3. 用户体系

### 3.1 注册与登录

- **注册**：昵称（唯一）+ 密码
- **密码存储**：bcrypt 哈希，不存明文
- **登录**：返回 JWT token，前端存 localStorage
- **Token 有效期**：7 天，过期需重新登录
- **昵称规则**：2-20 字符，支持中英文、数字、下划线

### 3.2 用户档案（Profile）

用户注册后可在设置页填写个人档案，所有人设共享：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `nickname` | string | ✅ | 注册时填写的昵称 |
| `age` | number | ❌ | 年龄 |
| `gender` | string | ❌ | 性别（男/女/其他/不愿透露） |
| `occupation` | string | ❌ | 职业 |
| `interests` | string[] | ❌ | 兴趣爱好，标签形式 |
| `bio` | string | ❌ | 自由文本，想让 AI 知道的其他信息 |

档案信息会注入每个人设的 system prompt，让 AI 从一开始就了解用户。

## 4. 记忆系统

### 4.1 双层记忆架构

```
┌─────────────────────────────────┐
│  用户档案 (Profile)              │  ← 用户自己填，所有人设共享
│  年龄、职业、爱好...              │
├─────────────────────────────────┤
│  人设专属记忆 (Persona Memory)   │  ← LLM 自动提取，每个人设独立
│  健身教练：膝伤、体重75kg         │
│  宠物专家：养柯基、3岁            │
└─────────────────────────────────┘
```

### 4.2 记忆提取机制

**触发时机**：每 **5 轮** 对话后触发一次记忆提取

**提取流程**：
1. 第 5/10/15/... 轮回复完成后，异步触发提取
2. 将最近 5 轮对话 + 现有记忆发给 LLM
3. LLM 返回需要新增/更新/删除的记忆条目
4. 合并写入记忆文件

**提取 Prompt 设计**：
```
你是一个信息提取助手。根据以下对话内容，提取用户透露的关键个人信息。

已有记忆：
{existing_memory}

最近对话：
{recent_messages}

要求：
1. 只提取用户明确透露的事实性信息
2. 与当前人设（{persona_name}）相关的信息优先
3. 如有冲突，以新信息为准（用户可能更新了情况）
4. 返回 JSON 格式：
{
  "add": {"key": "value", ...},      // 新增记忆
  "update": {"key": "new_value", ...}, // 更新已有记忆
  "remove": ["key1", ...]             // 删除过时记忆
}
5. 如果没有值得记住的新信息，返回空：{"add": {}, "update": {}, "remove": []}
```

### 4.3 记忆注入

每次对话时，system prompt 按以下顺序注入：
1. 人设性格设定
2. 用户档案（通用信息）
3. 人设专属记忆（个性化信息）

### 4.4 记忆可见性

- 用户可在前端查看每个人设记住的所有信息
- 用户可手动编辑或删除任意记忆条目
- 提供"清空记忆"功能（确认后执行）

## 5. 对话系统

### 5.1 对话模型

- **单会话制**：每个用户×每个人设维护一个连续对话
- **上下文窗口**：最近 **20 轮** 对话（用户+AI各算一轮）
- **流式回复**：SSE（Server-Sent Events），逐字输出

### 5.2 上下文构建顺序

```
1. System Prompt（人设设定 + 用户档案 + 人设记忆）
2. 历史消息（最近 20 轮）
3. 当前用户输入
```

### 5.3 对话存储

- 所有消息持久化到文件系统
- 每个用户×人设一个 JSON 文件
- 超过 200 条消息时，自动归档旧消息（保留记忆不丢）

### 5.4 清空对话

- 用户可清空某个人设的对话历史
- 清空对话 **不会** 清空记忆（记忆需要单独管理）


## 7. 安全与隔离

### 7.1 认证

- 所有 API（除注册/登录）必须携带 JWT token
- Token 中包含 username，后端校验请求的 user 与 token 匹配
- 防止用户 A 通过修改参数访问用户 B 的数据

### 7.2 数据隔离

- 文件系统按用户目录物理隔离
- API 层强制校验：`token.username == request.user_id`

### 7.3 并发安全

- 文件读写使用 asyncio.Lock，防止并发写坏 JSON
- 每个用户×人设粒度一把锁，不同用户不互相阻塞

## 8. 前端页面

### 8.1 页面清单

| 页面 | 路径 | 说明 |
|---|---|---|
| 登录/注册 | `/login` | 昵称+密码，支持切换登录/注册模式 |

### 8.2 响应式设计

- 移动端优先（to C 用户大多手机访问）
- 单页应用（SPA），纯前端路由
- 保持现有的简洁风格

## 10. 与 persona-chat 的关系

Nous 是 persona-chat Demo 的**产品化升级**：

| 维度 | persona-chat (Demo) | Nous (产品) |
|---|---|---|
| 用户 | 随机ID，无认证 | 注册登录，持久身份 |
| 记忆 | Hermes 巡检手动更新 | LLM 自动提取 + 用户可编辑 |
| 数据 | 混在一起 | 按用户严格隔离 |
| 上下文 | 全量历史 | 20轮窗口 + 记忆 |
| 前端 | 单页Demo | 多页面完整应用 |

persona-chat 保持不动，作为 Demo 参考。Nous 新建项目独立开发。

以下功能明确 **不在** 第一版范围内，但架构设计时需兼容：

- 自定义人设（用户自建 AI 角色）
- 多会话支持（每个人设多个对话线程）
- 语音输入/输出
- 图片理解
- 深度层（Hermes 巡检补充回复）
- 数据库迁移（文件 → SQLite/PostgreSQL）
- 微信登录 / 手机号登录
- 用户数据导出
